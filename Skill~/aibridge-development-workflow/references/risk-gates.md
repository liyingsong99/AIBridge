# 需求确认与风险门控

## 无需确认

满足以下条件时可以直接实施，并在 `【需求确认模式】` 中说明无需确认：

1. 用户目标明确，验收结果可以从上下文推断。
2. 修改范围小，主要影响单个文件、局部逻辑或文档。
3. 不改变公共 API、序列化数据结构、配置格式、资源路径或用户可见行为。
4. 不删除资源、不迁移数据、不引入新依赖。

## 必须确认

出现任一情况时先给方案并等待确认：

1. 用户需求存在多种合理理解。
2. 需要新增依赖、改变包结构、迁移配置或修改公共协议。
3. 会影响多个系统、公共 API、Prefab/场景结构或已有资源引用。
4. 涉及删除、覆盖、批量写入、不可逆修改。
5. 用户明确要求“先分析”“先给方案”“不要直接改”。

## 复杂度评估

- **常规问题**：已有模式清晰、影响范围有限，直接实施。
- **复杂问题**：涉及架构决策、跨模块影响、兼容性风险，先确认方案。
- **非常规问题**：现有模式不适用或需要创造性设计，先给多个方案和取舍。

## 输出建议

```text
【需求确认模式】
方案概述：...

待确认事项：
1. ...
2. ...

请确认后继续。
```
