AIBridge Workflow 设计与证据校验

这页把 workflow 的入口、业务模式、phase/step、artifact、gate、Verdict 和 Skill 交接放在同一张图里。 目标是让查阅者一眼分清：AI 先怎么路由、在哪些地方执行、什么时候交接，以及什么证据才算真正成立。

入口：Preflight / Skill 路由 执行：Mode Enter / Mode Execute 收尾：Mode Exit / SkillHandoff

阅读顺序

  1. 先看整体链路，确认入口和交接边界。
  2. 再看需求讨论和方案写入策略，确认什么时候先停下来讨论。
  3. 然后看“模式 vs 步骤”，分清业务目标和执行单元。
  4. 最后看证据链和 gate，确认什么能通过，什么只能算风险。

整体链路

workflow 的主线不是“把所有概念一次讲完”，而是先路由，再执行，再交接，再重新路由。 Preflight / Skill 路由 只负责选路和收缩上下文，不是业务模式。

01 Entry

Preflight / Skill 路由

  • 计算 baselineSkills、activeSkills、deferredSkills、guardedSkills
  • 先选主分支，再读分支文档
  • 不承担业务结果
02 Discuss

需求讨论模式

  • 只在目标、边界、验收标准或方案方向不清晰时触发
  • 先收敛需求，再决定是否写入方案文档
  • 确认后再回到正式路由
03 Enter

Mode Enter

  • 激活当前模式真正需要的 Skills
  • 只加载必要的 branch / checklist / helper 规则
  • 进入业务模式之前先把边界说清楚
04 Execute

Mode Execute

  • 按 recipe 的 phase/step 执行
  • 收集命令结果、artifact、日志、截图、Verdict
  • 以证据推进，而不是以感觉推进
05 Exit

Mode Exit / SkillHandoff

  • 输出 SkillHandoff
  • 释放下一模式不再需要的 Skills
  • 保留 artifact refs、gates、open risks
06 Loop

Transition Preflight

  • 用最新证据重新路由
  • 如果需要，切到下一个业务模式
  • 保持任务闭环，而不是一次性硬塞完
关键点： 入口只负责“怎么进来”，需求讨论负责“先把问题说清楚”，模式负责“现在干什么”，步骤负责“这一下做什么”。

模式 vs 步骤

模式 是面向任务目标的业务视角，步骤 是 recipe 里可执行的单元。 两者都重要，但职责完全不同。

模式

  • 决定 AI 现在是需求讨论、实施、调试、审查、验证，还是编排。
  • 决定当前需要哪些 Skills，以及何时交接。
  • 关注目标、风险、范围和输出形态。

步骤

  • 是 recipe 里的执行单元，通常落在 phase 下面。
  • 常见 kind：cli、agent、manual、barrier、report。
  • 关注单次命令、单次分析、单次交接或单次汇总。
维度 模式 步骤
职责 组织业务目标和技能范围 执行一个明确动作
位置 入口之后、收尾之前 recipe 的 phase 内
变化频率 低，通常按任务类型切换 高，按阶段连续推进
输出 SkillHandoff、风险、下一建议模式 ValidationResult、Finding、Verdict、ArtifactRef
典型例子 实施、调试诊断、审查、验证、编排 compile unity、runtime logs、verify-findings、write-report
常见误区： 把 Preflight 当作模式，或者把 step 当作完整任务。前者会把路由和业务混在一起，后者会把局部动作误认为最终结论。

方案写入策略

方案确认后，应先写入 `.aibridge/plan` 作为工作底稿，再按需要同步到正式文档目录。HTML 不是替代品，而是更适合流程图、决策树、对比表和开发者浏览的展示层。

场景 默认输出 说明
需要上下文恢复、subagent / multi-agent 续跑 .aibridge/plan/<slug>.md 作为工作底稿和状态锚点，方便压缩后回读。
短方案、纯文本约定、局部修订 .aibridge/plan/<slug>.md 优先保持可审阅、可 diff、可追踪；确认后更新为 confirmed。
有流程图、决策树、比较表 .aibridge/plan/<slug>.md + html html 作为工作底稿目录内的展示页；正式文档目录如存在则同步生成。
项目存在对应功能文档归类 .aibridge/plan/<slug>.md + 正式文档 工作底稿先落 `.aibridge/plan`，再同步到正式文档目录，便于 Git review。
用户明确希望给开发者看 .aibridge/plan/<slug>.md + html html 更适合浏览；若存在正式文档位置，再同步镜像。
推荐策略： 默认把 .aibridge/plan 作为方案工作底稿，正式文档目录作为发布版；这样 Git 里好 review，subagent 也有稳定回读点。

AI 的执行思路

这条链路的目的不是“多做一点”，而是“每一步都知道为什么做、做到哪、凭什么算成立”。

1

先路由

读取项目偏好和分支选择规则，确定当前主分支和需要加载的 Skills。此时重点是边界，不是改动。

2

再收基线

按任务类型收集编译、日志、Runtime、截图或语义证据，先把现状摆平，再往前走。

3

按步骤执行

单个 step 只做一件事。cli 跑确定性命令，agent 和 manual 通过外部执行器或人工完成，barrier 和 report 负责合并和收口。

4

再校验

把结果放到 gate 里验，不是放在口头判断里验。能复现、能引用、能导入，才算证据链的一部分。

5

最后交接

通过 SkillHandoff 说明已经完成的模式、释放了什么、下一步建议什么，以及还剩哪些风险。

严格证据校验

证据链的顺序是：command -> artifact / result -> gate -> Verdict -> report。 只要中间一环断了，就不能把结论说成已通过。

command / action 产生可复现操作
→
artifact / result 保存日志、截图、JSON、文件
→
gate 按规则判断是否满足
→
Verdict confirmed / refuted / uncertain
→
report 输出可审阅结论
严格规则： uncertain 不是通过，只是证据还不够。 workflow import 只接受结构化 JSON，不接受自由文本当作最终证据。
Gate kind 依赖的证据 判定方式 说明
unityCompile compile unity 命令成功 Unity 编译是否通过。
consoleErrors get_logs --logType Error 错误数 <= threshold.max 用于 Editor Console 基线校验。
runtimeReachable runtime status / runtime ping / runtime list_targets 命令成功 确认 Runtime/Player 目标能连上。
runtimeErrors runtime logs 错误数 <= threshold.max 用于 Runtime 错误计数。
screenshotExists 截图 artifact 文件 文件存在 不是“看起来像截图”，而是文件真实存在。
artifactRequired artifactKind / schema / stepId 过滤结果 数量 >= min 要求某类 artifact 真正落盘。
externalVerdict 导入的 Verdict artifact 允许的 status 命中 默认只接受 confirmed；uncertain 仍是缺口。
patchProposalRequired PatchProposal artifact 数量 >= min 用于要求先出方案再改动。
证据优先级建议： 真实命令结果和落盘 artifact 优先于总结性文字；结构化 import 优先于口头描述；引用 artifact id / command id 优先于复制大段日志。

Recipe 结构

workflow recipe 不是单个脚本，而是一个声明式的执行图。它先规定结构，再规定证据，再规定交接。

Recipe 顶层字段

常见字段如下：

  • schemaVersion, name, title, description, version
  • inputs, requiredSkills, phases, gates, artifacts

阶段 / 步骤类型

  • phase: serial, parallel, pipeline, barrier, report
  • step: cli, agent, manual, barrier, report
step kind 谁来执行 用途
cli AIBridge CLI 确定性命令，适合编译、日志、Runtime、截图、导入导出。
agent 外部 AI 执行器或主 agent 分析、审查、验证、生成 Finding / Verdict / PatchProposal。
manual 人工或主 agent 现场操作 点击、拖拽、人工确认、外部动作序列。
barrier CLI 的轻量汇总或主 agent 合并、去重、投票、归一化。
report CLI 的轻量汇总或主 agent 生成最终人类可读报告。

典型 recipe

下面这些例子把“模式”落实成“步骤序列”。它们的价值不在于名字，而在于每个阶段都知道该产什么证据。

实施

analyze -> implement -> validate -> report

调试诊断

frame -> collect -> reproduce -> analyze -> verify -> report

审查

shard -> parallel-review -> deduplicate -> verify -> report

验证

prepare -> perform -> collect-evidence -> verdict -> report

交接与释放

releaseSkillsAfter 和 SkillHandoff 解决的不是“删掉什么”，而是“下一步还需要什么”。 释放是作用域收缩，不是把已经读过的上下文物理清空。

SkillHandoff 常见字段

  • completedMode
  • releasedSkills
  • nextRecommendedSkills
  • summary
  • artifactRefs
  • gates
  • openRisks

交接时应该说清楚什么

  • 当前已经完成了哪个模式。
  • 哪些 Skills 可以放下，下一步建议重新加载什么。
  • 哪些 artifact 和 gate 已经成立，哪些风险还没关掉。
